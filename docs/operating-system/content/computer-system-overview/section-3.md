<h1><center>第三节 系统调用、运行映像、系统结构、引导与虚拟机</center></h1>

## 1. 操作系统接口

- **命令接口**：联机命令接口和脱机命令接口。
- **程序接口**：由一组系统调用组成，供应用程序请求内核服务。
- **图形接口**：通过窗口、图标和菜单等完成交互。

库函数不等于系统调用。库函数可以完全在用户态完成，也可能在内部封装系统调用。

## 2. 系统调用过程

1. 用户程序准备系统调用号和参数。
2. 执行陷入指令，由用户态切换到内核态。
3. 内核根据调用号执行相应服务例程。
4. 写回返回值，恢复用户程序现场。
5. 执行中断返回，回到用户态。

常见系统调用包括进程控制、文件读写、设备管理、内存申请和进程通信。

## 3. 程序运行时内存映像与地址空间

程序经编译、链接和装入后形成进程的运行时内存映像。典型用户地址空间从低地址到高地址包括：

- 代码段：保存可执行指令，通常只读。
- 只读数据段和数据段：保存常量、已初始化及未初始化的全局数据。
- 堆：由低地址向高地址动态增长。
- 共享库和内存映射区域。
- 栈：保存函数调用现场和局部变量，通常由高地址向低地址增长。

进程使用的是虚拟地址空间；地址转换机构把虚拟地址映射为物理地址，并实现进程隔离与保护。用户态通常不能直接访问内核地址空间。

## 4. 操作系统结构

### 4.1 分层结构

系统划分为若干层，上层只依赖相邻下层。优点是清晰、易验证，缺点是层次划分困难，跨层调用可能降低效率。

### 4.2 模块化结构

按功能拆分模块，模块间通过清晰接口协作。便于扩展，但模块依赖关系可能复杂。

### 4.3 宏内核

进程、内存、文件和设备等主要功能都运行在内核态。性能高，但内核庞大，故障影响范围大。

### 4.4 微内核

内核只保留最基本的机制，如进程通信、基本调度和低级内存管理，其他服务运行在用户态。

优点是可靠、易扩展、便于移植；缺点是服务之间频繁通信和模式切换可能产生额外开销。

### 4.5 外核

外核只负责安全地分配、回收和保护硬件资源，把较多资源管理策略交给用户态的库操作系统。它能让应用按需定制资源管理方式，但编程和兼容性更复杂。

## 5. 操作系统引导

典型引导过程为：

1. CPU 复位后执行固件程序，完成基本硬件检查。
2. 固件选择启动设备并装入引导程序。
3. 引导程序把操作系统内核装入内存并移交控制权。
4. 内核初始化中断、内存管理和设备驱动，创建首批系统进程。
5. 系统启动用户态服务并进入可用状态。

## 6. 虚拟机

虚拟机通过虚拟机监控程序把一台物理机抽象为多个相互隔离的执行环境。

- 第一类虚拟机监控程序直接运行在硬件上。
- 第二类虚拟机监控程序运行在宿主操作系统之上。

系统虚拟机可以运行完整的客户操作系统；同一物理机上的虚拟机共享硬件，但地址空间和设备状态彼此隔离。

## 7. 机制与策略

- **机制**回答“如何做”，如提供上下文切换能力。
- **策略**回答“做什么、何时做”，如选择短作业优先调度。

将机制与策略分离有利于复用和调整系统行为。

::: tip 判断方法
看到“应用程序请求操作系统服务”，优先想到系统调用；看到“普通函数计算”，不能直接断言发生了用户态到内核态的切换。
:::
